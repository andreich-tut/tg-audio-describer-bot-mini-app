# Project Context: tg-audio-describer-bot-mini-app

## Project Overview

A **Telegram Mini App** for audio transcription and AI-powered note-taking. This React-based frontend integrates with a backend API to provide voice transcription, AI chat responses, and Obsidian markdown note generation. The app is designed to work within Telegram's mobile interface with native-like UX.

### Core Features
- **Audio Transcription Modes**: Chat (transcription + AI response), Transcribe (raw text), Note (structured Obsidian markdown)
- **LLM Configuration**: Custom API endpoint, model selection, and API key management
- **Cloud Integration**: Yandex Disk OAuth integration for vault storage
- **Obsidian Sync**: Configurable vault paths and inbox folders
- **Telegram Native UX**: Haptic feedback, theme integration, and back button handling

## API Integration

### Auto-Generated Client (Orval)
The project uses [Orval](https://orval.dev) to generate type-safe React Query hooks from the backend's OpenAPI specification.

**Generated Structure:**
- `src/api/generated/models/` - TypeScript types from OpenAPI schemas
- `src/api/generated/{tag}/` - React Query hooks split by OpenAPI tags (settings, usage, oauth)
- `src/api/generated/request.ts` - Custom fetcher with Telegram authentication

**Usage Example:**
```tsx
import { useGetUsageApiV1UsageGet } from '@/api/generated/usage'

function MyComponent() {
  const { data, isLoading } = useGetUsageApiV1UsageGet()
  // ...
}
```

### Manual API Wrappers
Some APIs have manual wrappers for complex logic:
- `useSettings` - Settings CRUD with mutation helpers
- `useTelegram` - Telegram WebApp SDK integration (haptics, theme, back button)
- `useOAuthSSE` - OAuth connection state via Server-Sent Events

## Development Conventions

### TypeScript
- Strict mode enabled via `tsconfig.app.json`
- Path alias `@/` resolves to `./src/`
- Separate configs for app code and Node.js tooling

### ESLint
- Base: `eslint-plugin-react-hooks`, `eslint-plugin-react-refresh`
- TypeScript: `typescript-eslint` recommended rules
- Config: `eslint.config.js` (flat config format)

### React Patterns
- **React Compiler enabled** - Automatic memoization, avoid manual `useMemo`/`useCallback`
- **React Query for server state** - All API calls use `useQuery`/`useMutation`
- **Component composition** - Screen wrappers, tabbed navigation, view-based routing
- **Telegram-native UX** - Haptic feedback on interactions, native back button handling

### Styling
- Tailwind CSS v4 with custom design tokens
- Dark theme focused (slate/zinc palette with violet accents)
- Custom CSS in `theme.css` for animations and Telegram-specific adjustments

## Key Architectural Decisions

1. **View-based Navigation**: Simple state-driven routing (`currentView` state) instead of React Router
2. **Tab System**: Two main tabs (Mode, Settings) with bottom navigation
3. **Settings as Source of Truth**: All configuration stored in backend, synced via React Query
4. **OAuth via SSE**: Real-time OAuth connection status via Server-Sent Events
5. **Generated API Client**: Orval ensures types stay in sync with backend OpenAPI spec

## Telegram Mini App Specifics

- Uses `window.Telegram.WebApp` SDK for native integration
- Haptic feedback on all user interactions (mode selection, navigation)
- Theme-aware styling (currently dark-focused)
- Back button integration closes the mini app
- Full-screen mode via `tg.expand()`

### Architecture — FSD-lite (pages + shared)

The app uses a feature-sliced design lite structure. Organize code by feature/domain:

```
src/
├── api/
│   ├── generated/       # Auto-generated by Orval (DO NOT EDIT)
│   ├── client.ts        # Base API client
│   └── *.ts             # Manual API wrappers
├── features/            # Feature-sliced modules (one folder per feature)
│   ├── ModeSelection/   # Mode dashboard, mode switching logic
│   │   ├── index.ts     # Public API (barrel export)
│   │   ├── ui/          # Feature-specific components
│   │   └── model/       # Hooks, state logic owned by this feature
│   └── Settings/        # Settings screens, form logic
│       ├── index.ts
│       ├── ui/          # SettingsScreen, LLMConfig, VaultConfig, etc.
│       └── model/       # useSettings, form hooks
├── shared/              # Cross-feature only
│   ├── ui/              # Pure primitives (Button, Input, Badge, etc.)
│   │   └── (button, input, badge, card, screen-wrapper)
│   └── layout/          # Layout components, navigation
│       └── BottomNav.tsx
├── hooks/               # Global hooks (Telegram, OAuth SSE)
├── lib/                 # Pure functions only, no hooks
├── types.ts             # Global TypeScript types
├── App.tsx              # Root component with navigation state
├── main.tsx             # Entry point
└── theme.css            # Custom theme styles
```

#### Component Folder Pattern (Recursive)
Any complex component gets its own folder:
```
ComponentName/
├── index.tsx        # Public API
├── types.ts         # (if needed)
├── ui/              # Sub-components, only imported by this index.tsx
└── model/           # Hooks owned by this component
```

#### Rules
- `features/` = Feature modules that own their UI and logic
- `shared/ui/` = Pure primitives + shared cross-feature components. Must not import from features/
- `shared/layout/` = Navigation, layout wrappers
- `lib/` = Pure functions only, no hooks
- `hooks/` = Global app-level hooks (Telegram SDK, OAuth SSE)
- A hook lives in `model/` of the component that **calls** it first

### Context Structure — Folder Pattern
Contexts use the folder pattern with barrel exports:

```
src/contexts/
  LanguageContext/
    index.ts            # Barrel: re-exports everything
    LanguageProvider.tsx
    languageContext.ts  # createContext + types
    useLanguage.ts      # Hook
```

Apply this pattern to any new context added to the project.
